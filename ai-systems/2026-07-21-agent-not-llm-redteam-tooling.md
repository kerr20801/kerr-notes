# Agent 架構比模型選擇更重要——一次紅隊掃描工具的實測記錄

## 結論先講

這篇要講的不是「哪個模型比較強」，是一個更容易被忽略的事：**讓 AI 真正找到東西的，是它身邊那個「偵察 → 挑動作 → 執行 → 看結果 → 決定下一步」的迴圈，不是模型本身。** 同一顆模型，接上這個迴圈跟只是被單純問一句話，產出的價值天差地遠。

這篇記錄一次真實的重構過程：把一支既有的紅隊 AI 掃描引擎，從「手刻 prompt + 正則解析模型輸出」改成「原生 tool-calling」，過程中意外發現一個一直沒被用到的能力、一個關於「模型會不會拒答」的誠實觀察，以及一個不算完美收尾的限制。

## 起因

同事拿一般的 agentic coding 工具（掛在對話式 CLI 上、能自己寫腳本自己跑）配一顆本地開源模型，掃出了產品的真實漏洞。我們想知道：公司內部原本就有的紅隊 AI 掃描引擎，能不能做出類似的效果，而且不要每次跑都在燒雲端 API 的錢。

先把問題拆開來看：我們原本的引擎其實**已經是本地模型驅動的**，不是雲端 API 的問題。真正的差異在別的地方——引擎跟模型溝通的方式，是把整套「能力清單 + 已執行歷史」塞進一個超長的文字 prompt，要求模型輸出一個嚴格格式的 JSON，然後自己寫一堆正則表達式去從模型的回覆裡（常常夾雜了一堆思考過程的散文）硬撈出這個 JSON。撈不到就重新問一次，換句話說明格式，最多重試三次。

這是能動，但很脆弱——模型只要多說一句話、JSON 少個括號、思考過程把輸出額度用光，整條鏈就斷了，還得靠人事後寫防禦性程式碼去補洞。

## 意外發現：能力其實一直都在，只是沒被用到

現代的模型推論框架（vLLM、Ollama 都算）大多已經支援標準化的 **tool-calling**：你把「有哪些工具可以用、每個工具要吃什麼參數」用結構化的 schema 告訴模型，模型直接回一個結構化的「我要呼叫哪個工具、參數是什麼」，不用再靠正則表達式去猜。

查了一下才發現，我們正在跑的推論服務，啟動指令裡其實早就帶了開啟 tool-calling 的旗標——只是舊的引擎程式碼從來沒有用它，一直堅持用手刻解析那條路，可能是舊引擎剛寫出來的時候，這個能力還沒有那麼成熟/沒被驗證過穩不穩。

實測了一輪：拿同一顆模型，分別用「原生 tool-calling」跟「手刻解析」兩種方式問同一個問題（模擬一次真實掃描動作），tool-calling 版本正確判斷要呼叫哪個工具、參數填得對，而且看到工具回傳結果後，能正確整理成一段連貫、可用的回應，還會主動問要不要繼續往下測。這條路是通的。

## 重構原則：只換一層,其餘全部重用

寫新引擎的時候，刻意不重新發明輪子。原本的引擎裡：

- **能力清單**（工具描述 + 何時該用）是一份乾淨的資料結構
- **執行器**（每個能力實際怎麼打、怎麼判斷成功失敗）已經內建了扎實的證據紀律——例如「寫入類動作要先讀、改、寫、再讀回確認、最後還原」這種模式，避免對正式環境造成破壞
- **覆蓋率邏輯**（已知的攻擊面測完才准收尾，不能讓模型自己說「沒事了」就結束）

這三塊全部原封不動重用，新引擎唯一換掉的，是「模型怎麼決定下一步」這一層——把手刻的 prompt+正則解析，換成標準 tool-calling 呼叫。介面設計成跟舊引擎完全一致（同樣回傳「發現了什麼、執行過程、為什麼停下來」），這樣呼叫端可以直接互換比較，不用重寫任何上層邏輯。

## 實測結果

在一個授權的內部測試環境上跑了兩輪：

**未認證掃描**：全程正常運作，沒有找到東西——這是合理的，因為多數真正有價值的問題都需要登入權限才碰得到。這一輪主要驗證整條鏈路本身不會出錯、不會卡死。

**帶測試帳密掃描**：新引擎重現了一個已知的真實漏洞——一個設定寫入端點缺乏授權檢查，可以被用來竄改內部設定，執行器完整跑了「寫入 → 讀回確認 → 還原」的流程，證據扎實，測完也沒有在環境裡留下垃圾資料。

同時也踩到一個真的會讓輸出開天窗的坑：新模型的思考過程比舊模型冗長很多，如果生成長度上限沿用舊設定，複雜一點的題目會在模型還沒開始寫答案前，思考過程就把額度用光——整個回應變成空字串，狀態卻顯示「正常完成」，看起來像沒問題，其實是啞巴了。把生成長度上限調高後就正常了，但這提醒了一件事：**換模型不能只複製貼上舊參數，冗長度、輸出風格都可能不一樣，每個關鍵參數都要重新驗證，不能想當然爾**。

## 誠實講一個沒解決的限制

想進一步確認另一個已知問題（一種越權存取的漏洞類型）的時候，卡住了——因為手上能用的測試帳號權限層級不夠低，用高權限帳號測「能不能讀到別人的資料」本來就無法證明什麼（高權限帳號本來就看得到全部資料），執行器自己也誠實地把這次判定標成「證據不足」，沒有硬湊一個「找到了」。這條線目前沒辦法再往下確認，是已知的測試侷限，不是工具的 bug。

還有一個更根本的觀察：模型在某些特定的敏感技術類別（例如涉及身分系統的攻擊手法），就算 system prompt 已經明確講清楚這是已授權的內部測試情境，還是會先端出一段法律免責聲明，然後把原本該給的具體實作，退化成只教你用官方工具的用法。試過強化 prompt 去蓋掉這個行為，效果有限但沒有完全解決。

這點值得特別澄清一下：這**不是**某個特定模型來源地區的特性（不分中美，各家主流模型在「攻擊性工具產生」這個窄類別，都是重點加強防守的地方），純粹是模型本身在這個類別的對齊比較深，不是 prompt 工程能完全解決的事。

## 收尾：兩個不同的槓桿，不要混為一談

把這次的經驗濃縮成一句話：**Agent 架構決定了「探得多深、多穩」；模型本身決定了「肯不肯把話講完」。** 前者是工程問題，好好設計迴圈、重用扎實的執行邏輯就能穩定進步；後者是模型訓練層面的事，不是換一個更好的 prompt 就能徹底解決。

如果你也在做類似的事——用 AI agent 做需要多輪決策、多輪工具呼叫的自動化任務——這個區分可能比你以為的更重要：先把「迴圈」設計對，再去煩惱「該用哪顆模型」，順序不能顛倒。
