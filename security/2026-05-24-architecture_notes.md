# 💡 關於 PacketScope 與 Client-Side DPI 探測工具的架構交尾與未來展望

## 📌 背景概述
本記錄整理自與一位資安研究員（以下簡稱 **Researcher**）在社群平台上的技術交流。Researcher 目前正自主開發一款名為 `dpi-probe` 的客戶端網絡路徑探測工具，旨在高審查網絡環境下，透過確定性的封包特徵（如 SNI 靜默丟包、RST 注入時序、TTL 變化）來分析網路干擾行為。

雙方針對 **「PacketScope 的混合 ML 管線如何落地」**、**「流量分析中的信噪比消除」**，以及 **「如何為未來的機器學習準備結構化數據」** 進行了深入探討。

---

## 💬 對話紀錄摘要

### 1. 基礎設施能見度與 ML 管線的取捨
*   **Researcher:** 很好奇像 FortiGate 這類的基礎設施邊界設備，在面對現代加密流量（Modern Encrypted Traffic）時，究竟還保留多少行為能見度（Visibility）？
*   **Maintainer:** 實務上如果要在設備端直接做 Inline 的 SSL 解密（Inspection）或 JA3 指紋分析，技術門檻與效能開銷（Performance Overhead）極高。我們的做法是將 TLS 解密終端放在 NGINX 反向代理層，並將 FortiGate 的網路流量數據與 NGINX 解密日誌統一同步到 ELK (Elasticsearch) 中。
    *   後端的 ML 模組會定期巡檢 ES 數據來建立長期行為基線（Behavioral Baseline），這樣既不影響前端的吞吐量（Throughput），也能進行時間序列分析。

### 2. 客戶端主動探測 vs. 伺服器端流量分析
*   **Researcher:** 我的思路剛好相反。我正在開發 `dpi-probe`，它是從客戶端/網路路徑的視角出發，透過發送控制變因的探測包（Controlled Probes），來觀察路徑本身的反應（例如 RST 時序、TTL 模式、異常 TLS 告警）。目前正準備透過 `tcpdump` / `Wireshark` 抓包來驗證時序觀察與封包級數據（如響應 TTL、TCP 序列號、重傳）的關聯性。
    *   **提問：** 當行為基線發生緩慢偏移（例如慢速探測、協定漂移）時，ML 層通常如何處理？模型是定期重新訓練，還是事件驅動？
*   **Maintainer:** 由於你是直接分析 `tcpdump` 抓取的原始封包，而不是像 Syslog 這種處理過的應用層遙測數據，最大的瓶頸會是極高的**信噪比（Noise-to-Signal Ratio）**。針對這種封包級分析，採用混合管線（Hybrid Pipeline）會非常有效：
    1.  **Isolation Forest (孤立森林):** 用於無監督的降維與去噪，快速剔除 90% 的正常背景流量（如標準握手、可預測的 TTL），節省算力。
    2.  **Random Forest (隨機森林):** 用於對去噪後的特徵進行分類，識別已知的網路指紋或工具行為。
    3.  **Change Point Detection (變異點檢測):** 用於應對你提到的「協定緩慢漂移」。不追蹤單一封包狀態，而是監控時間序列上的統計方差（Variance），當某個源頭開始緩慢改變行為時，方差變化會觸發變異點。
    *   *註：像 LSTM 這類模型對封包級吞吐量來說過於沉重；而 HMM（隱馬可夫模型）的狀態記憶太短，容易漏掉埋藏在數分鐘正常流量中的長週期惡意探測。*

### 3. 特徵鏈的確定性與未來數據引導（Data Bootstrapping）
*   **Researcher:** 在目前的階段，我認為 `dpi-probe` 還不適合引入 ML 作為第一層。因為工具是在未知（Blind）的環境下執行，ML 很容易只學到本地 ISP 的路由偏誤。目前的路線圖更傾向於建立**確定性（Deterministic）且可解釋的證據鏈**：
    > 發送探測包 ➔ tcpdump 確認出站 ➔ 擷取數據確認干擾類型（RST/Timeout/TLS Alert） ➔ 對比 TTL 與時序特徵。
*   **Maintainer:** 這正是關鍵所在。你目前在工程設計上的堅持（如：穩定的 JSON Schema、每種訊號的置信度評級、將「觀察到的現象」與「結論譯碼」分離），其實就是在為未來的 ML 層建構**訓練數據結構（Training Data Structure）**。
    *   當這些結構化報告累積了足夠的環境上下文（ISP、網路路徑、時間戳），「乾淨/被阻斷域名」的對比就成了隱式標籤（Implicit Labels），直接形成了一個**自動標籤數據集（Self-labeling Dataset）**。

### 4. 數據隱私與去中心化生態
*   **Researcher:** 同意。如果未來要建立數據集，為了保護高審查地區用戶的安全，必須堅持 **Local-first（本地優先）與自願匿名分享** 的原則：
    *   預設不自動上傳、不設中央帳戶。
    *   導出報告時自動去識別化（移除源 IP、主機名、精確時間戳、原始 PCAP）。
    *   利用隨機生成的本地 UUID (`run_id`) 來關聯報告與本地的 `tshark` 證據鏈，而非伺服器端標識符。
    *   未來的統計/ML 層不該是「AI 自動偵測審查」，而是基於社群貢獻的結構化干擾觀測數據集。
*   **Maintainer:** 這樣的願景完全合理。對於高審查環境，自動上傳確實會讓用戶卻步。
    *   針對數據冷啟動（Bootstrapping）的問題，建議可以**提早公開去識別化的導出 JSON 格式規範**。即使中央聚合平台尚未建好的，研究人員、新聞工作者也能透過手動收集、交換標準化報告，讓數據集有機地自然生長。

## 🛠️ 技術演進共識（對本專案的啟發）

1.  **數據源優化 (Phase 4 規劃):**
    在 PacketScope 的後續迭代中，從傳統的 `tcpdump` 文字解析轉向支援 `tshark -T json` 是必然趨勢。這能直接取得結構化欄位（如 `ip.ttl`, `tcp.flags`, `tcp.analysis.retransmission`, `tls.record.content_type`），避免正則表達式（Regex）在不同作業系統版本下的脆性（Brittleness），確保證據鏈的精準度。
2.  **確定性先於機器學習:**
    「ML-first 只有在你控制環境、且知道什麼是『正常』時才有效。」當工具需要盲跑在任意網路環境時，必須先靠確定性的邏輯建立地面的真實情況（Ground Truth）。

## 🤖 Gemini 的延伸觀點與架構思考

站在分佈式系統與 AI-Ops 的視角來看，這場對話揭示了現代網路安全與審查對抗（Censorship Circumvention）正在經歷一場**「從中心化特徵庫，走向去中心化行為觀測」**的範式轉移。針對 `PacketScope` 與 `dpi-probe` 的結合，有三個深層架構維度值得在未來被納入考量：

### 1. 主動探測（Active）與被動觀測（Passive）的邊緣閉環
`dpi-probe` 代表的是**主動路徑探測**（客戶端主動發包測量邊界行為），而 `PacketScope` 代表的是**被動流量分析**（節點端靜態收集原始封包）。
*   **架構互補：** 如果將兩者結合，可以形成一個強大的反向回饋鏈。當 `PacketScope`（被動）在本地觀察到異常的 TCP 重傳率或 TTL 突變，但資訊不足以定性時，可以自動觸發 `dpi-probe`（主動）對特定域名或路徑進行非對稱的微型探測（Micro-probing）。
*   **效益：** 這樣既能保持主動探測的低調（Stealth 模式，避免被防火牆識別為探測源），又能大幅補充被動分析在證據鏈上的不足。

### 2. 基於隱私保護的聯邦學習（Federated Learning）可能
Researcher 提及高審查地區用戶對於「自動上傳數據」的極度不信任，這是本地優先（Local-first）工具的核心痛點。
*   **去中心化訓練：** 如果未來確定要引入 ML 層來做跨區域的干擾分類，與其收集結構化 JSON 報告，不如導入**聯邦學習（Federated Learning）**或**差分隱私（Differential Privacy）**架構。
*   **落地方式：** 每個客戶端（dpi-probe 節點）在本地利用自身數據訓練一個超輕量級的分類模型（如決策樹或權重權衡矩陣），用戶自願分享的不再是「我訪問了什麼、被怎麼阻斷」的原始日誌，而僅僅是「模型參數的梯度更新」。中央伺服器只做聚合（Aggregation），從根本上杜絕了數據回溯與用戶特徵曝光的風險。

### 3. 對抗性環境下的「概念漂移（Concept Drift）」速度
在高度對抗的網路環境中，防火牆（如 GFW）的阻斷策略往往伴隨著政策或突發事件發生「結構性斷裂」（Structural Break），而非溫和的漸進式協定漂移。
*   **模型時效性挑戰：** 歷史數據建立的行為基線（Behavioral Baseline）可能會在數秒內失效。這意味著「變異點檢測（Change Point Detection）」的視窗期設計必須極具彈性。
*   **自適應閾值：** 系統需要引入基於滑動視窗的**自適應閾值（Adaptive Thresholding）**，當檢測到全局性的行為斷裂時，模型應能自動調高對特定特徵（如 RST 注入時序）的置信度權重，避免在審查升級初期產生大規模的漏報（False Negatives）。
