# NGINX AI Shield 實戰偵測記錄：13 個請求內封鎖

## 前提

不求完美防禦，求提高攻擊者成本。

完美防禦不存在。一個防禦系統的價值不在於「能不能擋住所有攻擊」，而在於「讓攻擊者付出多少代價才能繼續」。這是設計 Shield 時的核心想法。

---

## 測試情境

授權黑盒測試，從外部視角，使用偵察手法探測一個部署了 NGINX AI Shield 的服務。
沒有任何內部知識，完全當成陌生目標。

---

## Shield 實際偵測到什麼

### 觸發的行為特徵

**1. session_phase_transition**

攻擊者通常有一個熱身過程：先瀏覽幾個正常頁面建立「正常 session」，然後突然切換到 API 探測或敏感路徑掃描。

Shield 偵測到這個轉換：session 從靜態資源存取突然切換到 API 呼叫。這個模式在正常用戶行為裡極少見。

**2. low_slow_probe**

慢速多路徑探測——把掃描請求分散在時間軸上、每次打不同路徑，目的是繞過基於請求速率的封鎖（傳統 rate limit 只看單位時間請求數）。

Shield 不只看速率，看行為模式。18 個請求、多路徑、低速，整體特徵符合低速掃描。

### 封鎖時間線

```
第 13 個請求 → 觸發封鎖
全部後續請求 → 403，無法繼續測試
```

攻擊者在取得任何有效情報之前就被擋住。

## Progressive Ban：不給第二次機會

這是讓我覺得設計對了的地方。

```
2026-05-20  → 偵測到掃描行為，狀態：alerted（記錄但未封鎖）
2026-05-23  → 同一來源再次觸發
              → 直接 banned（不再給觀察期）
```

第一次：給你一次機會，也許是誤報。
第二次：直接封鎖，不解釋。

正常用戶不會從同一個 IP 在幾天後再次觸發偵察特徵。這個模式幾乎只有兩種可能：攻擊者回來了，或者自動化掃描工具。兩種都應該封鎖。

---

## 攻擊成本不對稱

這次測試裡，沒有 Shield 的服務和有 Shield 的服務面對同樣的偵察手法，結果完全不同：

| | 無防護 | 有 Shield |
|--|--------|-----------|
| 攻擊者取得 API 地圖 | 是 | 否 |
| 攻擊者被偵測到 | 否 | 是 |
| 攻擊者留下記錄 | 否 | 是（threats.db）|
| 攻擊者能繼續 | 是 | 否 |
| 攻擊成本 | 極低 | 極高 |

Shield 不是讓服務「無法被攻擊」，是讓「攻擊變得不划算」。

攻擊者面對這個選擇：花更多資源換 IP、繞過偵測、重新建立 session——還是去找更容易的目標。大多數攻擊者會選後者。

---

## Shield 還不完美的地方

- 誤報率：行為偵測有時會封到正常的自動化工具（監控、CI/CD 健康檢查）
- Ban 到期後的重置：1 小時後解封，有組織的攻擊者知道等就好
- 沒有 IP reputation 整合：第一次出現的惡意 IP 要靠行為偵測，沒有歷史資料加速判斷

這些是已知的，不是意外。設計取捨是：寧可有誤報，不要讓真實攻擊靜默通過。
