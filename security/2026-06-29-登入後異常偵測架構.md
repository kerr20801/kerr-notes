# 登入後異常偵測架構

## Token 盜用 / Session 劫持防禦

> **適用場景**：客戶登入後，攻擊者竊取 Token 後模擬合法用戶大量取走資料。WAF 無法偵測此類攻擊，必須在應用層實作。


---

## 為什麼 WAF 不夠

| 防禦層 | 能做什麼 | 不能做什麼 |
|-----|------|-------|
| FortiGate | 封已知惡意 IP、DDoS 防護 | 看不懂 HTTP 內容 |
| WAF / nginx-shield | 擋惡意 payload（SQLi/XSS）、未認證掃描 | 不知道 token 是誰的、不知道行為是否異常 |
| **應用層** | **知道 token 屬於誰、該用戶正常行為是什麼** | —     |

攻擊者用偷來的合法 Token 登入，所有請求對 WAF 來說都是「正常的 200 OK」，無法區分。


---

## 威脅鏈

```
攻擊者竊 Token（XSS / 釣魚 / MitM / 惡意軟體）
    ↓
用合法 Token 登入 → WAF 完全看不出來（Token 有效）
    ↓
自動化腳本大量 GET /api/data、/export、/download
    ↓
資料外洩完成
```


---

## 推薦架構：三層，由淺入深

### Layer 0：規則層（先做，2 天內可上線）

純邏輯判斷，不需要 ML，抓最明顯的 case，命中立刻強制 re-auth 或封 Token：

```python
HARD_RULES = [
    # 同一 Token 同時從 2 個 IP 使用 → 最強信號，直接封
    concurrent_ip_count(token) > 1,

    # 10 分鐘內跨國家（不可能的旅行）
    geo_distance(prev_ip, curr_ip) > 500  # km
    and time_delta < timedelta(minutes=10),

    # 請求速率 >> 該用戶歷史基準的 5 倍
    req_per_min > user_baseline_rpm * 5,

    # 5 分鐘內 response bytes 超過 50MB（大量取資料）
    bytes_returned_5min > 50_000_000,
]
```

**觸發動作**：強制 Token 失效 + 發 Redis 事件給 LAP（見下方接口說明）


---

### Layer 1：ECOD 無監督異常偵測（第一個月用這層）

> **為什麼先用 ECOD 不用 XGBoost**：XGB 需要有標籤的訓練資料（哪些 Session 是被盜的）。系統剛上線時沒有這個資料。ECOD 不需要標籤，只需要正常 Session 資料就能建 baseline，適合冷啟動。

#### 每 5 分鐘對每個活躍 Session 計算 Feature Vector

```python
features = {
    "req_per_min":       float,  # 請求速率（最近 5 分鐘）
    "unique_endpoints":  int,    # 存取了幾個不同 endpoint
    "bytes_returned":    float,  # 取走多少資料（bytes）
    "unique_ips":        int,    # 這個 Token 從幾個 IP 打進來
    "hour_of_day":       int,    # 幾點（用戶凌晨 3 點的基準不同）
    "endpoint_entropy":  float,  # Shannon entropy（隨機亂打 vs 正常導覽）
    "export_api_ratio":  float,  # /export /download 佔總請求比例
    "new_ip_for_user":   bool,   # 這個 IP 這個用戶以前有沒有用過
    "time_since_login":  float,  # 登入幾分鐘了（秒）
}
```

#### 訓練方式

```python
from pyod.models.ecod import ECOD
import numpy as np

# 收集 2~4 週的正常 Session feature vectors
X_normal = load_normal_sessions()  # shape: (n_sessions, 9)

model = ECOD()
model.fit(X_normal)

# 儲存模型
import joblib
joblib.dump(model, "session_anomaly_ecod.pkl")
```

#### 線上推論

```python
model = joblib.load("session_anomaly_ecod.pkl")

for session in active_sessions:
    fv = compute_features(session)          # 計算 feature vector
    score = model.decision_function([fv])[0] # outlier score

    if score > THRESHOLD_WARN:   # 例如 0.7
        alert(session, score, "session_anomaly")
    if score > THRESHOLD_BLOCK:  # 例如 0.9
        invalidate_token(session.token_id)
```

### Layer 2：XGBoost 有監督分類（2\~3 個月後加）

等 ECOD 告警 + 人工確認積累了數百個正負樣本後，再訓 XGBoost。

**優勢**：`feature_importances_` 可以解釋「為什麼這個 Session 被標記」，比 ECOD 更可解釋，適合後續審計與調參。

```python
from xgboost import XGBClassifier

# 標籤：0 = 正常，1 = 被盜/異常
X_train, y_train = load_labeled_sessions()

clf = XGBClassifier(
    n_estimators=200,
    max_depth=4,
    learning_rate=0.05,
    scale_pos_weight=len(y_train[y_train==0]) / len(y_train[y_train==1]),  # 處理不平衡
    eval_metric="aucpr",
)
clf.fit(X_train, y_train)

# 推論
prob = clf.predict_proba([fv])[0][1]  # 被盜機率 0~1
```

## 上線時程建議

| 時程  | 工作  |
|-----|-----|
| 第 1 週 | Layer 0 規則層上線，接 Redis → LAP 告警 |
| 第 2\~4 週 | App middleware 記錄 session features → 存 DB |
| 第 1 個月底 | 正常 Session 資料夠了，訓第一個 ECOD model |
| 第 2\~3 個月 | 人工確認標籤累積，訓 XGBoost，兩層並跑比較效果 |

## 與 LAP 的接口

應用層任何一層命中，發一個 Redis 事件，LAP 接手後續動作：

```python
import redis

r = redis.Redis(host="127.0.0.1", port=6379)

r.xadd("lap.events", {
    "source":       "webapp",
    "type":         "session_anomaly",
    "severity":     "3",           # 1~5，建議 3=中危 / 4=高危
    "user_id":      user_id,
    "token_id":     token_id,
    "ip":           request_ip,
    "score":        str(anomaly_score),
    "trigger":      "concurrent_ip",  # 觸發的規則/模型名稱
    "action_hint":  "invalidate_token",
})
```

**Local AI 收到後自動執行**：

* FortiGate 封鎖該外部 IP
* Telegram 即時告警（含用戶 ID、觸發原因、分數）
* 開 escalation 工單（附完整 session 快照）
* 若 App 提供 `/admin/invalidate-token` API → 直接呼叫強制下線

## 重要注意事項

* **ECOD 冷啟動期**（前 2\~4 週）：先只收集資料，不跑偵測，避免正常 Session 被誤標
* **不要用 IP rate limit 替代 token rate limit**：攻擊者用你的 token，IP 看起來就是正常客戶
* **Response bytes 是最重要的特徵**：大量取資料一定反映在 bytes_returned，比請求數更可靠
* **hour_of_day 要做 per-user baseline**，不是全域基準（有人習慣深夜工作）
* **XGB 的 scale_pos_weight 一定要設**：被盜 Session 比例極低，不平衡資料不處理 XGB 會全部猜正常
