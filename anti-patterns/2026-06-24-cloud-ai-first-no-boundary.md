# 「先串雲的，成熟了再換地端」——聽起來很合理，錯在哪？

> **結論先說**：「雲優先」本身不是問題。問題是在沒有定義資料邊界之前就說「先串雲的」，等於把一個需要主管回答的問題，默默轉嫁給工程師自己判斷。工程師在沒有 policy 的情況下，只能做最快的決策，那個決策通常是錯的。

---

## 場景還原

很多公司在推 AI 導入時，主管會說這樣一句話：

> 「我們先用雲端 LLM 跑起來，熟練了再考慮串自建的地端模型。」

聽起來非常務實：降低初期成本、先驗證效果、循序漸進。

然後工程師拿到這個方向，去找「最快能串的方式」。

最快的方式是什麼？把現有系統的 API 直接接上去。於是 ticket 系統、log 系統、內部工單、運維記錄，開始流進雲端 LLM 的 context window。

---

## 主管的邏輯鏈斷在哪裡

「先串雲的」這句話預設了一件事：**瓶頸是技術能力，不是資料邊界。**

也就是說，主管認為「我們不熟地端模型的架構」才是問題，雲端串起來是為了讓團隊先有感覺，之後再換。

但實際上，大多數公司在 AI 串接上的第一個問題根本不是技術，而是：

**哪些資料可以出邊界？哪些不行？**

這個問題沒有人回答，工程師就只能根據「能不能跑起來」來判斷，不會根據「應不應該出去」來判斷——因為後者需要公司層級的資料分類政策，不是工程師能自己定的。

---

## 工程師做了「理性的錯誤決策」

這裡有一個容易被忽略的細節：

工程師通常沒有惡意。他在沒有任何邊界定義的情況下，做了最合理的選擇：**讓它跑起來**。

主管說「先串雲的」，沒說哪些資料不能串。工程師不可能主動去問一個主管自己也沒想過的問題。結果就是：系統跑起來了，demo 很漂亮，資料也出去了。

這不是工程師的問題，是一個沒有護欄的授權把責任靜默轉移了。

**「讓你們去研究怎麼用」這句話，翻譯過來是：你們自己決定邊界在哪。**

---

## 「雲優先」的適用條件

不是說雲端 LLM 不能用。是說有些前提要先成立：

| 條件 | 說明 |
|------|------|
| 資料已分類 | 哪些是 public / internal / confidential / restricted，有明確定義 |
| 串接只走 public 或 internal | confidential 以上的資料不出邊界 |
| 廠商合約確認 | 資料是否被用於訓練、保留多久、怎麼刪除 |
| 有人負責邊界判斷 | 不是「工程師自己決定」，是有明確的 owner |

這四件事如果都沒有，「先串雲的」等於「先把資料送出去再說」。

---

## 一個更準確的建議順序

常見的思路：**能力先，邊界後。**

```
學會用雲端 LLM → 跑起來有效果 → 之後再考慮資料問題
```

比較好的順序：**邊界先，能力後。**

```
定義資料分類 → 決定哪些資料可以出邊界 → 用那些資料去串 → 再慢慢擴展
```

第二條路開頭比較慢，但不會在三個月後發現「我們有敏感資料跑到外面了，但不知道從什麼時候開始的」。

---

## 主管能做的一件事

不需要懂 LLM 架構，也不需要懂地端部署，只需要在說「先串雲的」之前，先問一句：

> **「哪些系統的資料，你們認為不應該出公司邊界？」**

這個問題丟給工程師，他們會想。有人問了，就有人負責回答。沒有人問，那個問題就不存在，直到它變成問題。
