# 從「用而用」到 Security AI 的基礎 — ELK 架構演進紀錄

> 這篇文章記錄一個真實的演進過程：從什麼都不懂就把 ELK 跑起來，到重構成能支撐 AI 分析的資安平台。沒有一開始就設計好的藍圖，只有一次次踩坑之後的調整。

---

## Phase 1：先跑起來再說

一切的起點很簡單——**需要一個地方收 log**。

當時選 ELK 的原因也很直接：開源、社群資源多、搜尋功能強。但對 ES 本身的架構幾乎不熟，文件雖然看了，但很多概念要踩過坑才真的懂。

第一版的架構：

- ES 單版本（7.x）
- **Master Node 和 Data Node 混在同一台** — 當時覺得節省資源，沒想太多
- Logstash 簡單收，什麼 log 都往裡面丟
- 目標：能查到 log 就好

這個階段用起來沒問題，系統跑著。真正的問題要到用量變大之後才浮現。

---

## Phase 2：量來了，架構撐不住

當 log 量開始成長，第一版的問題開始出現。

**Master Node 混用的代價：** ES 的 Master Node 負責 cluster 狀態管理——哪些 shard 在哪台、新增節點、rebalancing。這些工作本來就有 overhead，當同一台機器同時在處理大量資料查詢，Master 的 response time 開始不穩定，偶爾會有 cluster 狀態飄移的問題。

這時候才真正去理解 ES 的角色設計：

- **Dedicated Master Node**：只管 cluster，不碰資料
- **Data Node**：只存資料、處理查詢
- **兩者混用**：在小規模可以，量一大就是在用自己的穩定性換資源節省

問題確診，但修起來代價不小——這個架構已經有一定的資料量，不是說重建就能重建。

---

## Phase 3：重構 — 這次認真設計

決定重新建一套，這次升到 ES 8.x，同時把架構想清楚再動手。

**新架構的核心決策：**

```
Master Node × 3  ──────────────────────────────
  │  只做 cluster 協調，不存資料，不處理查詢     │
  │                                              │
Data Node × N  ─────────────────────────────────
  │  只存資料，處理搜尋                           │
  │                                              │
Hot Node / Warm Node（按需求分層）               │
  │  Hot：近期資料，高 I/O                        │
  │  Warm：較舊資料，低存取                       │
                                                 │
Logstash（按資料來源分開）───────────────────────
  │  各自的 pipeline，各自的清洗邏輯              │
  │  不同來源不共用 pipeline，避免互相影響         │
```

**為什麼 Master 要 3 台？**

ES 的 cluster 靠投票機制決定 Master，3 台保證在 1 台掛掉時仍有 quorum（2/3）。2 台反而比 1 台危險——因為 2 台只要 1 台掛，就無法達成 quorum，整個 cluster 進入保護模式。

**Logstash 為什麼分開收？**

不同資料來源的格式差異很大——防火牆 log、VPN log、IDS 告警、系統事件，每種都有自己的欄位結構和清洗邏輯。混在同一個 pipeline 裡，debug 困難，改一個會影響另一個。分開之後，每條 pipeline 獨立，問題好追，也好擴充。

---

## Phase 4：架構對了，資料還不對

節點分開、架構穩了，以為可以開始用了。

結果發現：**有資料，但不是想要的資料。**

Logstash 收進來的欄位雜、格式不一致、時區錯誤、重複欄位。用 ES 查詢時發現很多想過濾的條件根本找不到對應欄位，或是同一個概念在不同來源用不同欄位名稱存著。

這個階段花了不少時間在：

- **欄位標準化**：定義統一的欄位命名規範，不同來源的 log 轉換成同一套 schema
- **時區問題**：資料來源的時區不統一，ES 的 `date_histogram` 在 UTC 和本地時間之間的坑
- **Logstash filter 設計**：grok、mutate、date filter 的組合，把原始 log 清洗成可用的結構
- **`.keyword` 欄位**：ES 的 text field 預設不能做精確匹配，aggregation 要用 `.keyword`——這個坑踩過一次之後就記住了

這個階段的產出是一套比較穩定的資料 schema，往後所有的查詢和分析都建立在這個基礎上。

---

## Phase 5：資料對了，還是看不懂

有了乾淨、結構化的資料，新問題出現：**資料量太大，人看不完。**

每天進來的 log 量以百萬計，告警數量也不小。人工審視的效率遠遠跟不上。

這是引入 ML 的起點——不是為了追流行，而是**因為資料有了，但人的注意力是瓶頸**。

初期的 ML 應用很直接：
- 對流量行為建立基線，偏離基線的觸發告警
- 對攻擊特徵做分類，減少重複的低品質告警
- 把每天的告警做自動摘要，讓值班人員先看最重要的

效果是真的有：告警的 signal-to-noise ratio 明顯提升，值班人員開始有餘力深入調查，而不是每天在清告警佇列。

---

## 現在的樣子

經過幾個階段的演進，架構大致如下：

```
資料來源
  │ Firewall / VPN / IDS / System Event / ...
  ↓
Logstash（各來源獨立 pipeline）
  │ 清洗 → 標準化 → 時區統一
  ↓
Elasticsearch 8.x
  │ 3 Dedicated Master + Data Node（Hot/Warm 分層）
  ↓
ML 分析層
  │ 行為基線 / 異常評分 / 告警分類
  ↓
AI 應用層
  │ Security Chat（LLM + ES 查詢）
  │ 自動日報（LLM 產出摘要）
  │ 威脅情資研究工具
```

## 幾個回頭看的心得

**1. 「用而用」不是錯，但要知道它的代價**
第一版架構不好，但它讓我們先把資料收進來，驗證了這個方向是對的。如果一開始就等設計完美再動手，可能到現在還沒資料。代價是後來要花時間重構，但這個代價是值得的。

**2. 架構問題不會馬上爆，但一定會爆**
Master 混用在小規模完全沒問題，問題是量大之後才出現。這類問題很難預防，但可以在第一次重構時一次修好，不要留著。

**3. 資料品質比架構更重要**
架構穩了，但資料不乾淨，分析出來的結果還是垃圾。花在 Logstash pipeline 和欄位標準化的時間完全值得，這是之後所有 ML 和 AI 分析的地基。

**4. AI 是最後一層，不是第一層**
很多人想直接用 AI 解決資安問題，但 AI 需要乾淨的資料、穩定的架構。把基礎做好，AI 才能發揮作用。順序不能反。
