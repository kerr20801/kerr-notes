# 讓 AI 知道要查什麼 — LLM 與 ES 的 Security Chat 設計

> 第一篇談架構演進，這篇談核心設計：怎麼讓 LLM 理解資安問題、決定查哪些資料、再把結果說清楚。這中間有不少細節，也踩了不少坑。

---

## 問題的起點

資料有了，架構穩了，告警也在跑。但使用者還是得自己進 Kibana 寫 query 才能回答問題。

> 「過去一小時有哪些外部 IP 在掃我們的服務？」
> 「這個 IP 有沒有在 VPN 登入記錄裡出現過？」
> 「今天有沒有異常的大量資料傳出去？」

這些問題對熟悉 ES 的人來說是幾行 query 的事，但對大多數使用者來說是門檻。引入 LLM 的目標很明確：**讓不懂 ES query 的人也能直接問問題、得到答案。**

---

## 直覺做法的問題

最直覺的做法是把 ES 的資料塞給 LLM，讓它直接回答。

問題是：**ES 的資料量太大，根本塞不進 context window。**

即使塞得進去，把幾十萬筆 log 給 LLM 也毫無意義——它沒辦法在那麼多資料裡找出有意義的模式，而且成本極高、速度極慢。

正確的方向是反過來：**先讓 LLM 決定要查什麼，再去 ES 拿精確的資料，最後讓 LLM 解釋結果。**

---

## 核心架構：兩階段處理

```
使用者問題
    │
    ↓
【第一階段：Tool Calling】
LLM（推理模型）分析問題
    │ 判斷意圖：這個問題需要哪些資料？
    │ 選擇工具：呼叫對應的 ES 查詢函數
    │ 傳入參數：從問題中提取關鍵資訊（IP、時間範圍等）
    ↓
ES 查詢執行
    │ 拿到精確、有限的資料（不是全部 log）
    ↓
【第二階段：串流回答】
LLM（快速模型）根據查詢結果回答
    │ 有資料背景，能說清楚
    │ 串流輸出，使用者不需要等
    ↓
使用者看到答案
```

兩個階段，兩個不同的 LLM，各做各的事。

---

## 意圖分類：LLM 怎麼知道要查什麼

使用者的問題千變萬化，但背後需要的資料有限。把常見的資安問題整理成幾個意圖類別：

| 意圖 | 對應的問題類型 | 查的資料 |
|------|--------------|---------|
| `external_attack` | 外部有誰在攻擊我們？ | Firewall / IDS 外部告警 |
| `vpn` | 這個帳號的 VPN 登入情況 | VPN 認證 log |
| `internal_anomaly` | 內部有沒有異常行為？ | 內部流量異常 |
| `exfil` | 有沒有資料外洩的跡象？ | 大量外傳流量記錄 |
| `investigate_ip` | 這個 IP 是誰、做了什麼？ | 多來源 IP 行為 |
| `nas_users` | NAS 使用者異常存取 | 檔案存取 log |

LLM 在 tool calling 階段的工作就是：**從使用者的問句中判斷意圖，選對工具，抽出參數。**

例如：
```
使用者：「10.20.1.5 這個 IP 最近有沒有異常？」
LLM 判斷：investigate_ip
呼叫工具：get_ip_profile(ip="10.20.1.5", hours=24)
```

---

## 為什麼用兩個不同的 LLM？

這是整個架構裡最重要的一個決策，也是踩坑最多的地方。

**Tool calling 用推理模型（非串流）**

Tool calling 需要精確——模型要正確判斷意圖、正確選工具、正確抽參數。這件事需要比較強的推理能力，但不需要快，因為結果是給程式用的，不是直接給使用者看的。

用推理型的大模型，非串流呼叫，給它足夠的時間想清楚，輸出限制在 300 tokens 以內（因為只需要 function call 格式，不需要長篇大論）。

**串流回答用快速模型**

當 ES 查詢結果回來，要告訴使用者「結果是什麼、代表什麼意思」。這個階段需要的是**速度**，不是深度推理——使用者在等，不能讓他盯著空白螢幕。

用輕量快速的模型做串流輸出，使用者看到文字一行一行出現，體感好很多。

**為什麼不全用推理模型做串流？**

踩坑紀錄：把推理模型直接拿來做串流回答，使用者的螢幕會先空白 9-30 秒才開始出現文字。

原因是推理模型在輸出答案之前會先做大量內部推理（reasoning tokens），這段時間完全沒有輸出。對使用者來說，送出問題之後什麼都沒有，很容易以為壞掉了。

**結論：兩件事的需求不同，用不同的模型。**

## 工具函數的設計

每個工具函數負責把「LLM 的判斷」轉換成「ES 的查詢語言」。

以 `get_external_attacks` 為例：

```python
def get_external_attacks(hours: int = 24, severity: str = None) -> dict:
    """
    查詢外部攻擊告警
    hours: 查最近幾小時
    severity: HIGH / CRITICAL，不傳就查全部
    """
    query = {
        "bool": {
            "must": [
                {"range": {"@timestamp": {"gte": f"now-{hours}h"}}},
                {"term": {"direction": "inbound"}},
            ]
        }
    }
    if severity:
        query["bool"]["must"].append({"term": {"severity": severity}})

    result = es.search(index="security-alerts-*", query=query, size=100)
    # 整理成 LLM 容易理解的格式
    return summarize_attacks(result)
```

工具函數做三件事：
1. 接收 LLM 傳來的參數
2. 轉成 ES query
3. 把結果整理成結構化的摘要（不是把原始 log 全丟給 LLM）

**第三步很重要。** LLM 拿到的不是原始 log，而是「過去 24 小時有 15 個外部 IP 攻擊，其中 3 個是 Critical，最活躍的是 x.x.x.x，主要攻擊手法是 SQL Injection」這樣的摘要。這樣 LLM 才能說出有意義的分析，而不是重複貼 log。

## 一個完整的對話流程

```
使用者：「今天有沒有嚴重的外部攻擊？」

─── Tool Calling 階段（使用者看不到）───
LLM 分析：external_attack 意圖
選擇工具：get_external_attacks(hours=24, severity="HIGH")
ES 查詢：搜尋今日 HIGH/CRITICAL 外部告警
結果：發現 3 個 Critical，來自 2 個 IP

─── 串流回答階段（使用者看到）───
「今天偵測到 3 個嚴重等級的外部攻擊事件：
1. IP 203.x.x.x — 持續的 SQL Injection 嘗試，共 847 次請求，目標集中在登入頁...
2. ...
建議優先確認 203.x.x.x 是否已被封鎖，...」
```

使用者問了一個自然語言問題，得到一個帶有資料背景的具體分析，中間的 ES query 對他們完全透明。

---

## 幾個設計原則

**1. LLM 不應該直接接觸原始 log**

原始 log 格式雜、欄位多、很多對 LLM 沒意義的資訊。讓工具函數做好預處理，LLM 只拿摘要。

**2. Tool calling 失敗要有 fallback**

有些問題 LLM 判斷不了意圖，或是抽出的參數有問題。這時候要有合理的 fallback——要嗎直接回答（不查 ES），要嗎告訴使用者「這個問題需要更多資訊」。

**3. 串流的使用者體驗優先**

資安使用情境時間感很強。等 30 秒才看到答案，使用者體驗很差。串流回答讓使用者有「AI 在想」的感覺，而不是盯著空白等結果。

**4. 把模型選擇和業務邏輯分開**

模型會換，業務邏輯不應該跟著換。工具函數、意圖分類這些應該獨立，只有最後一層知道現在用哪個模型。
