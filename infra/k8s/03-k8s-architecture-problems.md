# K8s 架構為什麼會亂 — 從「為了用而用」說起

> 結論先說：K8s 亂的根本原因不是 CNI 選錯、不是 Ingress 設定問題，是在沒想清楚用途之前就建起來了。工具換來換去解決不了這個問題。

---

## 最常見的起點：為了用而用

很多團隊建 K8s 的起點是「我們應該要用 K8s」，不是「我們有這個需求，K8s 能解決」。

這個起點決定了後面的走向：

- 沒有核心用途，什麼都往裡面塞
- Stateful 服務（DB、ES、NAS）也進去了
- Pod 邊界沒想清楚，一個 Pod 塞太多，或切太細
- Namespace 沒規劃，全部丟 default

結果：K8s 跑著，但維運複雜度遠高於直接用 Docker Compose，而實際得到的好處很少。

遇到問題就換工具——CNI 從 Flannel 換到 Calico，Ingress 換了又換——但問題還在，因為根本原因不在工具。

---

## 三個問題判斷要不要用 K8s

**1. 這個服務需要水平擴展嗎？**

不需要的，不用進 K8s。一個內部工具跑一個 Pod，跟直接跑 Docker 差在哪裡？維運複雜度高很多，好處幾乎沒有。

**2. 這個服務是 Stateless 的嗎？**

有狀態的服務進 K8s 要付出很高的維運代價。DB、ES、大型 Queue 通常放外面比較好，讓 K8s 管 Stateless 的 App tier 就好。

**3. 團隊有沒有能力維護？**

K8s 壞掉要能查。沒有人懂 eBPF、沒有人會看 kube-apiserver log，出問題就是一場災難。工具的複雜度要配得上團隊的能力。

三個問題都沒有的話，Docker Compose 就夠了，而且出問題好查、新人容易接手。

---

## K8s 真正值得用的場景

**CI/CD Runner**

Runner 是 K8s 最天然的使用場景。動態起 Pod 跑 job、用完就刪、資源按需分配——這正是 K8s 的強項。多部門共用一個 Runner Pool，各自的 pipeline 在自己的 Namespace 跑，資源隔離、互不干擾。

**物件儲存（MinIO）**

多部門共用、需要擴展、MinIO 本身對 K8s 支援不錯。這種有明確需求的服務放進去合理。

**需要水平擴展的 App 服務**

流量有明顯高低峰、需要滾動更新、需要跨節點排程——這些情況 K8s 發揮的價值才對得起維運成本。

我們自己的環境就是以 Runner 和物件儲存為核心建起來的，其他服務是後來有需求才加進去，不是一開始就全塞。

---

## 不重建的情況下怎麼改善

很多環境已經亂了，但重建成本太高，沒辦法從頭來過。這種情況有兩個改善方向不需要大遷移：

**Namespace 先切好**

按部門或服務類型切 Namespace，立即效果：
- 各部門資源隔離，不互搶
- NetworkPolicy 邊界清楚，好寫好維護
- 出問題範圍縮小，不會一個壞全部受影響

這件事在現有環境上就能做，不需要動到服務本身。

**加 Runner，給 K8s 一個真正的用途**

搭配 Namespace 規劃，讓各部門的 CI/CD pipeline 跑在 K8s 上。這樣 K8s 就有了一個真正合理的核心用途，不再是「為了用而用」。

這兩件事做完，K8s 的存在就有了合理的理由，後續的調整也比較有方向。

---

## 如果要做遷移

如果環境亂到必須重整，幾個原則：

**先建後拆，不要在原環境動刀**

新架構建好、確認跑得起來，再把服務一個一個移過去，最後拆掉舊的。不要直接在生產環境改，風險太高。

**有 CI/CD 最好**

有 pipeline 的話，每次改動有紀錄、部署有驗證、出問題知道哪一版壞的。沒有的話每步都要手動確認，速度慢、風險高。

**中間狀態不能拖**

新舊並存、流量一半一半、維護兩套設定——這個狀態要盡快結束。拖太久，最後兩套都變成技術債。時間表要定清楚，不然半年後還在「遷移中」。

---

## 回頭看

K8s 不是萬能的，也不是越大越好。

架構大小不是重點，用途是不是真實需求才是重點。一個小而清楚的 K8s 環境，比一個大而亂的好維護太多。

真正的問題從來不是 CNI 選哪個、Ingress 用哪套。是「這個服務為什麼要跑在 K8s 上」這個問題，在建之前有沒有認真回答過。
