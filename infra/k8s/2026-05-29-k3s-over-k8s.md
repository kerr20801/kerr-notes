# 幾個 Pod 不需要 3 Master — 我怎麼看 K8s vs K3S 的選擇

寫於 2026-05-29

---

最近看到一個架構：幾個服務 Pod，撐起了 3 Master + 2 Node + 1 DB 的完整 K8s cluster。

前面兩台 Nginx 做 proxy，這個我沒意見，合理。

但後面那六台機器，我不會這樣做。

---

## 我的判斷框架

K8s 解決的是**大規模服務的調度、彈性伸縮、自癒**。

你的服務沒有這個規模需求，K8s 對你來說是純成本：

- etcd 三節點需要維護
- control plane 本身就吃資源
- 升級、憑證輪換、etcd 備份全是額外工作
- 出問題的 surface area 大很多

規模沒到，用 K8s 是在幫自己製造維運負擔。

---

## 我會怎麼做

**K3S，兩台，Keepalived HA。**

```
[ Nginx .8 ] [ Nginx .9 ]   ← 前面不動，繼續做 proxy
        ↓
    VIP (Keepalived)         ← .8 主、.9 備，VIP 自動漂移
        ↓
[ K3S Master .A ]            ← server mode，embedded etcd
[ K3S Master .B ]            ← server mode，embedded etcd
        ↓
  Pods 跑在這兩台上
```

Nginx 只需要把流量導到 Keepalived 的 VIP，不需要知道後面是哪台 K3S 在服務。

---

## K3S HA 兩種做法

### 做法一：兩台 K3S + Keepalived + DB 獨立

適合服務有獨立 DB 需求的場景。

```
K3S Master A ─┐
              ├── Keepalived VIP ← Nginx 導流
K3S Master B ─┘

DB A ←→ DB B（RSYNC 或 replication）
```

K3S 本身用 embedded etcd，兩台互為 HA。  
DB 另外兩台做同步，或直接用外部 managed DB。

### 做法二：兩台 K3S + Keepalived，DB 掛 PVC

適合服務簡單、DB 流量小的場景。

```
K3S Master A ─┐
              ├── Keepalived VIP
K3S Master B ─┘
      ↓
  DB Pod（掛 PVC，NFS 或本地 volume）
```

DB 跑在 K3S 裡，storage 用 PVC。簡單，但 DB Pod 漂移時要注意 volume 跟得上。

---

## 機器數對比

| 架構 | 機器數 | 維運複雜度 |
|------|--------|-----------|
| 原本 K8s（3M+2N+1DB）| 6 台 | 高 |
| K3S HA（2台 K3S）| 2 台 | 低 |
| K3S HA + 獨立 DB | 4 台 | 中 |

功能一樣，機器少一半，維運複雜度砍掉大半。

---

## 什麼時候才需要真的 K8s

- Node 數量超過 10 台
- 需要跨 AZ 部署
- 有專門的 Platform Engineering 團隊維護
- 已經在用 EKS / GKE / AKS 等 managed service

如果以上都不是你的情況，K3S 夠了。

但很多人會說以後一定會擴大所以...當然所有架構都是自己的選擇沒有一定。
